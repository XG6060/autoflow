## 一、项目愿景

### 一句话

**让任何人都能构建自动化工作流——按价值付费，不按搬运次数收费。**

### 核心命题

当前市场存在一个矛盾：Zapier 容易上手但贵且锁定你，n8n 能力强但学习曲线拒绝了非技术用户。两者之间的"甜区"——**易用 + 不锁定 + 不惩罚增长的定价**——完全空白。

### 长期愿景

不只是自动化工具，而是 **SMB 的运营中枢**：自动化 → 轻量CRM → 健康监测，三步吃掉 Zapier + HubSpot 的 SMB 市场。

## 三、产品定义

### 产品名称（待定）

**AutoFlow**（备选：FlowKit / OpenFlow / Pipekit）

### 核心差异化

| 维度 | Zapier | n8n | **AutoFlow** |
|------|--------|-----|-------------|
| 易用性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 能力强 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 定价友好度 | ⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 数据可移植 | ⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 主动告警 | ⭐ | ⭐ | ⭐⭐⭐⭐⭐ |

### MVP 功能范围

```
第一版交付（4个月内）：
├── 可视化工作流编辑器（拖拽 + 双向画布）
├── 15个核心集成
│   ├── Gmail / Outlook
│   ├── Slack
│   ├── Google Sheets
│   ├── Shopify
│   ├── Stripe
│   ├── HubSpot
│   ├── Notion
│   ├── Airtable
│   ├── Webhook (in/out)
│   ├── HTTP Request(万能兜底)
│   ├── Filter / Router / Delay
│   ├── Code Node (JS/Python)
│   └── Email (SMTP)
├── 主动健康检查 & 多渠道告警
├── 一键工作流导出 (YAML/JSON)
├── 执行历史 & 调试日志
└── 免费/付费方案
```

### 不做的事项（明确拒绝）

- ❌ 不做 8,000+ 集成（用 Webhook + HTTP + 开放社区弥补）
- ❌ 不做 AI Agent（V1 聚焦核心自动化）
- ❌ 不做移动端 App（V1）
- ❌ 不做自托管（V2+，V1 验证托管模式的 PMF）

### 与 Zapier 同档位对比

| 方案 | Zapier 月价 | **AutoFlow** | 节省 |
|------|------------|-------------|------|
| 基础 | $19.95（750任务） | $19（10,000执行） | 等价但容量多13× |
| 中型 | $69（5,000任务） | $79（100,000执行） | 等价但容量多20× |
| 大型 | $299+（50,000任务） | $299（1,000,000执行） | 等价但容量多20× |

> **关键差异**：AutoFlow按"工作流执行次数"计费，一个30步的工作流 = 1次执行。Zapier按"任务步骤数"计费，30步 = 30次任务。实际价差可达 **10-30倍**。

## 六、技术架构概览

### 技术选型

```
前端：    React + TypeScript + ReactFlow(画布) + Tailwind CSS
后端：    Node.js + TypeScript (API) + Go (执行引擎)
数据库：   PostgreSQL (元数据) + Redis (队列/缓存)
消息队列： BullMQ (基于Redis)
部署：    Docker + Kubernetes (托管) / Docker Compose (自托管V2)
监控：    Grafana + Prometheus
CI/CD：   GitHub Actions
```

### 架构图（文字版）

```
┌─────────────────────────────────────────────┐
│                  用户层                       │
│  Web App (React)  │  API Client  │  CLI      │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│               API Gateway                    │
│       (Auth / Rate Limit / Billing)          │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│              核心服务层                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
│  │ Workflow │ │ Execution│ │  Connector   │ │
│  │ Designer │ │  Engine  │ │   Registry   │ │
│  └──────────┘ └────┬─────┘ └──────────────┘ │
│                    │                         │
│  ┌─────────────────▼──────────────────────┐  │
│  │           Health Monitor               │  │
│  │  (异常检测 / 告警 / 自动重试)           │  │
│  └────────────────────────────────────────┘  │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│              数据 & 基础设施                  │
│  PostgreSQL │ Redis │ S3(日志) │ Docker/K8s │
└─────────────────────────────────────────────┘
```

### 为什么选择这些技术

| 选择 | 理由 |
|------|------|
| React + ReactFlow | 工作流编辑器的最佳Web技术栈，n8n也用Vue+自定义画布验证了这条路 |
| Go执行引擎 | 工作流执行对性能敏感，Go的并发模型天然适合，且编译为单二进制便于V2自托管 |
| Postgres + Redis | 成熟、托管服务便宜、运维成本低 |
| 先不做自托管 | MVP阶段验证PMF优先，自托管增加运维复杂度且降低收入 |
| 开放集成规范(JSON Schema) | 从Day 1就让第三方贡献集成，学习n8n社区节点模式 |

## 九、风险与应对

| # | 风险 | 概率 | 影响 | 应对 |
|---|------|------|------|------|
| 1 | **集成生态鸡生蛋问题**（用户需要X集成但你还没有） | 高 | 高 | HTTP节点万能兜底 + Day 1开放集成规范 + 社区贡献激励 |
| 2 | **n8n加速改善易用性**（AI辅助构建器等） | 中 | 高 | 速度是你的优势——4个月内出MVP，比大团队快 |
| 3 | **大厂降价**（Microsoft Power Automate等） | 低 | 中 | 大厂不会放弃按用户/按操作的计费模型，这是结构性优势 |
| 4 | **做出来了没人用**（冷启动失败） | 中 | 高 | Phase 0专门验证需求，如果等名单<20人 → No-Go |
| 5 | **托管成本过高**（免费用户消耗大量计算资源） | 中 | 中 | Free层硬限制(5个工作流/1000执行) + 执行超时(30秒) |
| 6 | **Zapier的集成数量绝对优势** | 高 | 中 | 不走"追平"策略——20个高频集成覆盖80%场景 + 万能HTTP兜底 |
